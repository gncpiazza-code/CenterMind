Cómo trabajamos: agentes IA en paralelo desde la terminal

Claude Code CLI + equipo de sub-agentes especializados corriendo en background simultáneamente

01

Plan de implementación

Se generó un implementation_plan.md exhaustivo con el análisis de cada ítem, las hipótesis de bug, los archivos a tocar y el orden de ejecución seguro para evitar conflictos.

02

Detección de dependencias

Se identificaron qué ítems comparten archivos (p.ej. TabSupervision.tsx era tocado por Item 2 y Item 3B simultáneamente). Se dividió el trabajo en ondas para evitar conflictos de escritura.

03

Wave 1: 4 agentes en paralelo

Los ítems sin archivos en común se lanzaron como agentes independientes en background. Mientras un agente tocaba api.py, otro modificaba el dashboard, otro el informe, y otro el sidebar. Sin bloqueos.

04

Wave 2: agente de supervisión

Una vez que Wave 1 terminó (y el backend ya tenía los nuevos campos), se lanzó el agente de supervisión que tocó los 3 archivos frontend interrelacionados: api.ts, TabSupervision y MapaRutas.

05

Recuperación de fallos

Dos agentes fallaron por saturación del API (error 529). El ítem simple (Sidebar) se resolvió directamente en la terminal principal. Los otros se re-lanzaron con prompts más acotados.

06

Commit único + push

Con todos los cambios listos, un commit atómico con los 10 archivos fue pusheado a main. Railway y Vercel detectaron el push automáticamente y desplegaron en ~2 minutos.

Item 1 — Bug: Cuentas Corrientes de Tabaco no mostraba vendedores

api.py · línea 3239

Al seleccionar la sucursal de Tabaco & Hnos en el panel de supervisión, la tabla de Cuentas Corrientes aparecía vacía. El endpoint devolvía 0 registros aunque cc_detalle tenía ~4.578 filas para ese distribuidor.

❌ Antes
q = q.eq("sucursal_nombre", sucursal) # Comparación exacta, case-sensitive # "CÓRDOBA" ≠ "Córdoba" ≠ "cordoba" # El selector mandaba "Córdoba" pero la DB # tenía "CÓRDOBA" → 0 resultados
✅ Después
q = q.ilike("sucursal_nombre", sucursal.strip()) # ilike = LIKE case-insensitive en Supabase # Ignora mayúsculas, tildes y espacios # "córdoba" = "CÓRDOBA" = "Córdoba " ✓

❌ Antes
// Un único bloque que traía todo const fetchAll = useCallback(async () => { setLoading(true) const [kpis, ranking, ...] = await Promise.all([...]) setKpis(kpis) setRanking(ranking) setLoading(false) }, [distId, año, mes, ...]) useEffect(() => { fetchAll() const id = setInterval(fetchAll, 90_000) return () => clearInterval(id) }, [fetchAll])
✅ Después

Beneficios clave

Sin flash vacío al filtrar
placeholderData mantiene los datos anteriores visibles mientras llega la nueva query.
Carga independiente por sección
KPIs y ranking aparecen en cuanto están listos, sin esperar al resto.
Auto-refresh sin setInterval
refetchInterval: 90_000 maneja el ciclo automáticamente con cleanup correcto.
Animación stagger Framer Motion
Los KPI cards entran en cascada con opacity + translateY en 300ms.

Item 6 — Fix: impresión del mapa en blanco (bug WebGL)

MapaRutas.tsx

El botón "Imprimir mapa" generaba una página en blanco. Causa: los navegadores no renderizan canvas WebGL en el contexto de impresión CSS. La solución captura el mapa como imagen PNG antes de imprimir.

❌ Antes — CSS print (no funciona con WebGL)
const handlePrint = () => { // Inyectaba estilos @media print // y llamaba window.print() // El canvas WebGL no se renderizaba // → hoja en blanco window.print() }
✅ Después — captura canvas primero
// MapaRutas init: new maplibregl.Map({ ..., preserveDrawingBuffer: true // nuevo }) const handlePrint = () => { const canvas = mapRef.current .getCanvas() const dataUrl = canvas.toDataURL('image/png') const win = window.open('','_blank') win.document.write( `<img src="${dataUrl}"> + leyenda` ) win.print() }

Deploy a producción

Railway (backend) + Vercel (frontend) — automático desde main

Backend — Railway

Push a main → Railway detecta el cambio → rebuild del contenedor → uvicorn reiniciado. Tiempo: ~90 segundos. Los fixes de api.py están activos en producción.

https://api.shelfycenter.com

Frontend — Vercel

Push a main → Vercel build → Next.js optimizado → CDN. TypeScript/ESLint errors no bloquean el build (ignoreBuildErrors: true). Tiempo: ~2 minutos.
